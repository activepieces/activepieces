For any outbound HTTP in `packages/server/{api,worker,utils}`, use `safeHttp.axios` or `safeHttp.createAxios({ ... })` from `@activepieces/server-utils`. These wrap `request-filtering-agent` to reject private, loopback, link-local, and cloud-metadata IPs (configurable via `AP_SSRF_ALLOW_LIST`).
Never call raw `fetch(...)` or `axios.create(...)` for URLs sourced from user input, admin config, OAuth token/refresh endpoints, or third-party integrations — they bypass SSRF protection and close the DNS-lookup-to-connect TOCTOU window the filtering agent handles.
Hardcoded calls to trusted Activepieces endpoints (`api.activepieces.com`, `secrets.activepieces.com`) should still use `apAxios`, which is itself built on `safeHttp`.

When a library needs a `fetch` function rather than axios (e.g. the MCP SDK transports), use `safeHttp.createSafeFetch()` from `@activepieces/server-utils` — NOT the global `fetch`. Node's built-in `fetch` (undici) ignores `http.Agent`, so passing it a `request-filtering-agent` does nothing and SSRF protection is silently lost. `createSafeFetch` is an adapter that runs over `node:http`/`https` with the same filtering agents `safeHttp.axios` uses (so private/loopback/link-local/metadata IPs are rejected at connect time) and returns a WHATWG `Response` with a streaming body. This matters most off the sandbox path — the worker process is not egress-proxied, so user-supplied URLs (MCP server URLs, etc.) reaching it are a real SSRF risk.
