# Core packages split thin (bundleable) from thick (`@activepieces/shared`)

`packages/core/*` is ordered thin → thick: `core-utils`, `core-piece-types`, `core-formula`, `core-execution` are framework-agnostic, dual-format, side-effect-free libraries that pieces and the engine may import; `core/shared` is the one thick, app-level member — it keeps the name `@activepieces/shared`, carries DB/EE/management schemas and heavy deps (`dayjs`, `expr-eval`, `socket.io-client`), and depends on the thin members. Pieces and the engine must **never** import `@activepieces/shared` — they get what they need re-exported through `@activepieces/pieces-framework` — because pulling the thick package into a piece bundle bloats it ~6× and drags in server-only code. The boundary is enforced per-package, not by folder name; see `.claude/rules/core-packages.md`.
