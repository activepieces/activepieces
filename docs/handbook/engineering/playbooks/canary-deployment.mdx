Canary deployment lets us run a separate app instance on a newer (or experimental) build and route a specific subset of platforms to it, while all other traffic continues hitting the primary app. This gives us real-world validation with live data before a full rollout.

## Architecture Overview

```
Browser / External Webhook
         │
         ▼
   Primary App (prod)
   ┌─────────────────────────────┐
   │  preHandler hook            │
   │  canaryRoutingMiddleware    │
   │    ├─ platformId ∈ canary?  │──── No ──▶ handle normally
   │    └─ Yes                   │
   │         ▼                   │
   │  HTTP proxy to canary app   │──────────▶ Canary App
   └─────────────────────────────┘               │
                                                  ▼
                                         Canary Worker
                                         (polls canaryWorkerJobs queue)
```

All traffic enters the primary app. The `canaryRoutingMiddleware` runs as a Fastify `preHandler` hook on every request. If the resolved `platformId` is in the canary list, the request is proxied in-process to the canary app and the response is returned directly to the caller — the primary app never processes it further.

## Scope: what canary can and cannot validate

<Warning>
  Canary is a **backend-only** environment. It is built to validate API, engine, worker, trigger, and webhook changes against live production data — **not** frontend changes, and **not** changes that alter the shape of data the frontend reads.
</Warning>

A canary-routed platform runs a **mixed-version stack**:

```
Cloud (current prod) frontend ──XHR──▶ Primary app ──proxy──▶ Canary backend (new build)
                                                                └─▶ Canary worker (new engine)
```

- **The UI is served from the production (cloud) deployment, never from canary.** The canary CD workflow deploys only the app (`config/app-canary.yml`) and workers (`config/worker-canary.yml`) — there is no canary frontend. The React SPA is a separate deployment behind `AP_FRONTEND_URL`; the API server does not serve it, and `canaryRoutingMiddleware` runs *after* authentication on API requests only, so it never proxies static assets. Canary users therefore load the **current prod frontend** and only their authenticated API calls reach the canary backend.
- **The engine is aligned with the backend.** Canary workers are deployed from the same image tag as the canary app and run the bundled engine from that build, and canary-platform jobs are routed only to the canary worker queue. So backend + engine are always the same version — this is the safe part.

### Implication: do not canary changes that migrate the flow data model

Flow-version migrations (`flowVersionMigrationService`) run **on read** and **persist forward** to the shared database — they bump `schemaVersion` and rewrite the stored `trigger`, with no inverse migration. Combined with the mixed-version stack, this creates two hazards:

1. **Frontend/backend skew.** If a PR bumps `LATEST_FLOW_SCHEMA_VERSION` *and* changes the builder UI (new step shapes, new expression/output format), the canary backend migrates a canary platform's flows to the new shape, but those users are served the **old** prod frontend, which cannot render the new shape correctly and may **corrupt the flow on save**. For example, a migration that rewrites `{{step_1.foo}}` into `{{step_1['output'].foo}}` will display as raw tokens in the old UI's data selector.
2. **Forward-only / rollback risk.** Once a canary platform's flow is migrated, the row is permanently at the new schema version. Moving the platform off canary, or rolling the canary image back (older engine, same migrated data), leaves the old code unable to execute it correctly. The pre-migration snapshot is kept in `backupFiles[<oldSchemaVersion>]`, but recovery is manual.

**Rule of thumb:** full-stack PRs and any PR that bumps `LATEST_FLOW_SCHEMA_VERSION` should ride a **full production promotion** (frontend + backend shipped together, short rolling-deploy skew window, roll-forward-only) rather than a backend-only canary. If such a change must go through canary, the migration must be gated off for canary platforms until the frontend they are served also understands the new shape.

## Canary membership: DB-backed with Redis cache

Canary membership is one case of a general **worker-group** model. A platform's group is stored in the `platform_plan.workerGroupId` column, and a platform is "canary" when `workerGroupId === 'canary'`. On each lookup, the group is fetched from Redis under a per-platform key (`platform:<platformId>:worker_group_id:v2`, 5-minute TTL; a `__none__` sentinel is cached for platforms with no group). On cache miss the value is read from the database and cached. When the group is changed via the API, that platform's cache key is invalidated immediately.

This removes the need to keep canary membership in sync across services and allows runtime changes without a redeploy.

<Note>
  The legacy `platform_plan.canary` boolean column is **deprecated** (removal planned in `0.83.0`) and was migrated into `workerGroupId` by the `MergeCanaryAndDedicatedWorkersIntoWorkerGroupId` migration (release `0.82.0`). The same migration folded the old `dedicatedWorkers` concept into `workerGroupId`, so any code or query touching canary membership should read `workerGroupId`, not `canary`.
</Note>

## Components

### Canary App

A second instance of the server API running a different image tag. No special env vars are needed for canary membership — it is driven entirely by the DB. Configure it with:

| Env var | Value |
|---|---|
| `AP_FRONTEND_URL` | canary app's public URL |

The canary app runs as a full app instance. Global queue consumers (`runsMetadata`, `system-job-queue`) run on both the primary and canary app — this is safe because BullMQ ensures each job is processed by exactly one consumer.

### Canary Workers

Dedicated workers that connect to the canary app instead of the primary app:

| Env var | Value |
|---|---|
| `AP_WORKER_GROUP_ID` | `canary` |

Workers use `AP_FRONTEND_URL` for their Socket.IO RPC channel and for posting engine results back to the app. Canary workers are registered with the canary app, so the RPC path is fully isolated.

### Primary App Config

The primary app only needs to know where to proxy canary requests:

| Env var | Description |
|---|---|
| `AP_CANARY_APP_URL` | Internal URL of the canary app |


## WebSocket / Real-time Updates

Socket.IO is configured with a **Redis adapter** (`@socket.io/redis-adapter`). Events emitted on any app instance (primary or canary) are broadcast through Redis pub/sub to all connected instances. This means:

- Users connected to the primary app receive real-time flow run updates even when the execution happened on the canary app.
- No WebSocket proxying is required.

## Queue Isolation

| Queue | Primary App | Canary App |
|---|---|---|
| `workerJobs` | ✅ Consumed by primary workers | Not consumed |
| `runsMetadata` | ✅ Consumed | ✅ Consumed |
| `system-job-queue` | ✅ Consumed | ✅ Consumed |
| `platform-canary-jobs` | Not consumed | ✅ Consumed by canary workers |

Jobs for canary platforms are enqueued to `platform-canary-jobs`, which only canary workers poll — fully isolated from the primary worker fleet. The queue name is derived per worker group as `platform-<workerGroupId>-jobs`, so the canary group (`workerGroupId = 'canary'`) maps to `platform-canary-jobs`.

When a platform is moved into or out of the canary group, any already-enqueued jobs are drained from the source queue to the target queue before the flag is flipped, so toggling canary never strands in-flight work.


## Deploying a New Canary Build

The **Continuous Delivery — Canary** workflow (`continuous-delivery-canary.yml`) runs automatically every day at 9 AM UTC, building from the latest `main`. It is also invoked as a reusable workflow from `continuous-delivery-cloud.yml` on the Sunday promotion, so the canary is always refreshed from the latest `main` immediately before production is updated.

The workflow:
1. Builds and pushes a new image tagged `<version>.<sha>.canary`
2. Checks if any new migrations are breaking — fails the workflow if they are (no override)
3. Deploys the canary app (`config/app-canary.yml`) and canary workers

Required GitHub secrets: `AP_API_KEY` (the primary app's `AP_API_KEY` value).

## Rolling Back a Canary Deployment

Trigger the **Continuous Delivery — Rollback Canary** workflow (`continuous-delivery-rollback-canary.yml`) with the image tag you want to roll back to. The workflow:

1. Extracts the migration manifest from the target image
2. Rolls back DB migrations not present in that manifest
3. Redeploys the canary app and workers to the target image

| Input | Description |
|---|---|
| `rollback_to_image_tag` | Image tag to roll back to (e.g. `0.51.0.abc1234.canary`) |
| `force` | Force rollback even if breaking migrations exist. Default: `false`. |

## Promoting canary build to production

Canary is a validation environment, not a promotion path. Full promotion happens via the Sunday scheduled cloud workflow, which deploys `release-candidate` to production. If a canary build has been validated and the corresponding commit has been tagged as `release-candidate`, it will automatically reach production on the next Sunday.

## Managing Platform Routing

### Enable canary routing for a platform

```bash
curl -X POST https://cloud.activepieces.com/v1/admin/platforms/canary \
  -H "api-key: <AP_API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"platformId": "<platformId>", "canary": true}'
```

### Disable canary routing for a platform

```bash
curl -X POST https://cloud.activepieces.com/v1/admin/platforms/canary \
  -H "api-key: <AP_API_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"platformId": "<platformId>", "canary": false}'
```

Both calls set `platform_plan.workerGroupId` (to `canary` or `null`) in the database, migrate any in-flight queued jobs to the target queue, and invalidate that platform's Redis cache key (`platform:<platformId>:worker_group_id:v2`). The `{ canary: true | false }` request body is kept for backwards compatibility — internally it writes `workerGroupId`. The change takes effect on the next request — no restart required.

<Tip>
  To route a platform to an arbitrary (non-canary) worker group, use the sibling endpoint `POST /v1/admin/platforms/worker-group` with a `{ platformId, workerGroupId }` body.
</Tip>
