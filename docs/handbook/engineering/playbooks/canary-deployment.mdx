Canary deployment lets us run a separate app instance on a newer (or experimental) build and route a specific subset of platforms to it, while all other traffic continues hitting the primary app. This gives us real-world validation with live data before a full rollout.

## Architecture Overview

```
Browser / External Webhook
         │
         ▼
   Primary App (prod)
   ┌─────────────────────────────┐
   │  preHandler hook            │
   │  canaryRoutingMiddleware    │
   │    ├─ platformId ∈ canary?  │──── No ──▶ handle normally
   │    └─ Yes                   │
   │         ▼                   │
   │  HTTP proxy to canary app   │──────────▶ Canary App
   └─────────────────────────────┘               │
                                                  ▼
                                         Canary Worker
                                         (polls platform-{id}-jobs queue)
```

All traffic enters the primary app. The `canaryRoutingMiddleware` runs as a Fastify `preHandler` hook on every request. If the resolved `platformId` is in the canary list, the request is proxied in-process to the canary app and the response is returned directly to the caller — the primary app never processes it further.

## Components

### Canary App

A second instance of the server API running a different image tag. It is configured with:

| Env var | Value |
|---|---|
| `AP_FRONTEND_URL` | canary app's public URL |

The canary app runs as a full app instance. Global queue consumers (`runsMetadata`, `system-job-queue`) run on both the primary and canary app — this is safe because BullMQ ensures each job is processed by exactly one consumer.

### Canary Workers

Dedicated workers that connect to the canary app instead of the primary app:

| Env var | Value |
|---|---|
| `AP_FRONTEND_URL` | canary app's URL |
| `AP_PLATFORM_ID_FOR_DEDICATED_WORKER` | platform ID this worker is dedicated to |

Workers use `AP_FRONTEND_URL` for their Socket.IO RPC channel and for posting engine results back to the app. Canary workers are registered with the canary app, so the RPC path is fully isolated.

### Primary App Config

The primary app needs to know which platforms to route and where to send them:

| Env var | Description |
|---|---|
| `AP_CANARY_APP_URL` | Internal URL of the canary app (e.g. `http://canary-app:3000`) |
| `AP_CANARY_PLATFORM_IDS` | Comma-separated list of platform IDs to route to canary |


## WebSocket / Real-time Updates

Socket.IO is configured with a **Redis adapter** (`@socket.io/redis-adapter`). Events emitted on any app instance (primary or canary) are broadcast through Redis pub/sub to all connected instances. This means:

- Users connected to the primary app receive real-time flow run updates even when the execution happened on the canary app.
- No WebSocket proxying is required.

## Queue Isolation

| Queue | Primary App | Canary App |
|---|---|---|
| `platform-{id}-jobs` | Not consumed (dedicated workers only) | Not consumed (dedicated workers only) |
| `runsMetadata` | ✅ Consumed | ✅ Consumed |
| `system-job-queue` | ✅ Consumed | ✅ Consumed |

Canary workers push jobs to `platform-{platformId}-jobs` which canary workers then pick up — fully isolated from the primary worker fleet.


## Adding a Platform to Canary

### 1. Enable dedicated workers for the platform

Call the admin endpoint on the primary app to enable dedicated workers and create the platform-specific BullMQ queue:

```bash
curl -X POST https://cloud.activepieces.com/api/v1/admin/platforms/dedicated-workers \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $AP_API_KEY" \
  -d '{
    "operation": "enable",
    "platformId": "<platformId>",
    "trustedEnvironment": false
  }'
```

### 2. Deploy a canary worker for the platform

Spawn a canary worker with `AP_PLATFORM_ID_FOR_DEDICATED_WORKER` set to the platform ID. The worker will connect to the canary app via `AP_FRONTEND_URL` and exclusively poll the `platform-{platformId}-jobs` queue.

### 3. Add the platform ID to the routing list

Update `AP_CANARY_PLATFORM_IDS` on the primary app — no restart of the canary app or workers is needed. The middleware reads the env var on every request.

```bash
# Single platform
AP_CANARY_PLATFORM_IDS=<platformId>

# Multiple platforms
AP_CANARY_PLATFORM_IDS=<platformId1>,<platformId2>
```

## Removing a Platform from Canary

1. Remove the platform ID from `AP_CANARY_PLATFORM_IDS` on the primary app — traffic immediately reverts to the primary app.
2. Shut down the canary worker for that platform.
3. Disable dedicated workers via the admin endpoint with `"operation": "disable"`.
